# 0018: 零停机滚动部署与容器名冲突自愈 (Zero-Downtime Deployment & Conflict Healing)

- **状态**: 接受
- **日期**: 2026-03-23

## 1. 背景上下文
在先前的版本中，为了解决 `docker compose up` 期间由于不同 Project Name 遗留容器导致的 `container name "zen70-xxx" is already in use` 冲突，部署脚本采用了暴力的 `docker rm -f $(docker ps -aq --filter name=zen70-)` 方案。
这种“先杀光后拉起”的模式会导致全栈服务产生哪怕是 10-30 秒的物理停机时间，严重违反了系统高可用（HA）要求以及零停机滚动升级的原则。

## 2. 决策选项
1. **方案 A**: 维持原样，容忍每次部署短暂停机。
2. **方案 B**: 引入复杂的蓝绿部署或金丝雀发布机制，利用宿主机上的额外负载均衡器（如 Traefik 等）切换流量。
3. **方案 C**: 回归 Docker Compose 原生语义，统一 Project Name 强声明，利用 `compose up -d --remove-orphans` 的默认增量滚动重载特性，并辅助以前置检测-摘除 (Detect-Adopt-Retry) 机制。

## 3. 评估对比
### 方案 A (维持现状)
- **劣势**: 完全背离系统的 SRE P99 可用性红线，暴力杀容器在挂载卷未解除占用时极易引发死锁。

### 方案 B (蓝绿部署)
- **劣势**: 引入巨大的运维成本和状态同步风险。对于单点拓扑的家庭服务器来说属于过度设计（Over-engineering），会成倍占用本就紧张的存储和内存资源。

### 方案 C (原生 Compose 滚动 + 冲突自愈)
- **优势**: 只需要对已有脚本进行轻度重构：消除对 `COMPOSE_PROJECT_NAME` 的随机分配或错误定值（如安装器的硬编码 `"32"` 回归 `"zen70"`），系统便可顺理成章地依靠 Compose 引擎完成新旧容器的无缝剔换。
并且通过自愈逻辑：在遇到跨 Project 残留容器干扰时，退化为只针对冲突节点精确下达 `docker rm -f` 及 `compose ... down` 指令。这在保证了 0 停机的同时，亦彻底根除了重名冲突。

## 4. 最终决定
采用 **方案 C**。
1. 完全废除部署过程中的全局 `docker rm -f` 操作。
2. 将启动命令彻底收束为：`docker compose up -d --remove-orphans`。
3. 于 Python 部署脚本层面实装 `Detect -> Adopt -> Retry` 逻辑以进行异常降级。

## 5. 影响范围
- SRE/Ops 流程变更：线上发版或重配热加载，将真正对终端请求完全透明，无缝衔接。
- 完全消除了部署引发的 502/503 报错风暴。


## 6. 交叉验证补充：SRE 生命探针防线的补齐 (2026-03-23)
单纯依靠 compose up -d --remove-orphans 执行平滑更新时，若缺乏服务真实就绪状态的判定，仍可能在短时出现上游请求转发至进程在但仍处于内存初始化的真空服务期。
在 V3.3 后期的 system.yaml 大规模更新中，已为全量 16 个核心服务追加了业务相关的深层 Healthcheck 及长达 15-30 秒不等的 stop_grace_period。
借此，Docker 引擎的替换时序彻底演变为：
1. 立项启动带后缀的新容器。
2. 内部原生探针（如 pg_isready / 
edis-cli ping）连续 3 次返回就绪。
3. 引擎将内部路由切至新节点。
4. 向老容器下发 SIGTERM，老容器靠 30s 宽限期优雅中止并发 TCP 会话、刷盘 WAL 事务。
5. 老容器自动销毁。
至此，本体系的零停机滚动演进到了工业级绝对无损标准。
